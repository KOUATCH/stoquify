### Execution Checklist

1. Start with a short mission-control note.
   - State what will be inspected.
   - State the success criteria.
   - State which agents own which lane.
   - State what will not be touched.

2. Inspect evidence before planning changes.
   - Read `AGENTS.md` if present.
   - Read `package.json`.
   - Read `config/sidebar.ts` and `config/permissions.ts`.
   - Inspect `graphify-out/GRAPH_REPORT.md` and any more specific graph outputs if present.
   - Inspect relevant `what-next/` reports, especially architecture, UI/UX, BI, payroll, module-control, service-boundary, and release-gate reports.
   - Inspect the route, service, action, hook, Prisma, and component files for the target area.

3. Produce the modernization map.
   - Current strengths.
   - Trust gaps.
   - UI consistency gaps.
   - BI/read-model gaps.
   - Security/privacy/compliance gaps.
   - Operational workflow gaps.
   - Performance and reliability risks.
   - Evidence missing before any readiness claim.

4. Build the prioritized roadmap.
   - P0: trust, tenant isolation, RBAC, module entitlement, financial correctness, payroll/compliance immutability, proof-backed BI, and route readiness.
   - P1: authenticated UX consistency, route taxonomy, action queues, evidence drawers, dashboard freshness, data contracts, and workflow recovery states.
   - P2: packaging, subscriptions, usage telemetry, customer success, growth loops, pricing, and advanced BI.
   - P3: optional innovation that has a clear customer outcome and does not weaken trust.

5. Execute in small verified slices.
   - Keep changes scoped to one workstream at a time unless write scopes are disjoint.
   - Prefer existing services, components, hooks, and primitives.
   - Add abstractions only when they remove real duplication or enforce an important invariant.
   - Preserve dirty-worktree changes that are not yours.
   - Never use destructive git or database reset commands unless explicitly requested.

6. Run the Dev -> QA -> Fix loop.
   - Developer or worker implements the slice.
   - Code Reviewer reviews the diff.
   - Security/AppSec reviews trust-boundary impact.
   - API Tester/Test Results Analyzer verify behavior.
   - Evidence Collector captures screenshots or JSON/markdown evidence where relevant.
   - Reality Checker decides whether the evidence is enough.
   - Retry up to 3 times for a failing slice, then report the blocker with exact evidence.

7. Save durable artifacts.
   - Save the main report under `what-next/agent-orchestration/`.
   - Save screenshot bundles under the relevant `what-next/<domain>/screenshots/` directory when UI is involved.
   - Save machine-readable JSON evidence when a gate or smoke test produces it.
   - Update docs only when they describe a real current behavior, decision, or runbook.

### Risk Controls

- Do not claim production readiness without current evidence.
- Do not fabricate financial, payroll, fiscal, provider, or compliance data.
- Do not move business truth into React components.
- Do not bypass tenant scope, RBAC, module entitlement, fresh auth, approval, or audit requirements.
- Do not expose raw provider references, payroll data, fiscal payloads, customer balances, or audit details to unauthorized UI.
- Do not hard-delete immutable financial, payroll, compliance, audit, close, reconciliation, or POS records.
- Do not normalize UI by inventing a new design system when existing command-center and BI primitives can be reused.
- Do not let package/subscription logic become RBAC. Entitlement is tenant-level; RBAC is user-level.
- Do not hide blocker states. Use partial, unavailable, locked, redacted, or stale states with clear reasons.
- Preserve existing user changes in the worktree.

### Success Criteria

The mission is successful only when:

- The agent team has produced evidence-grounded findings, not generic advice.
- The roadmap ties every proposed improvement to a user role, business outcome, risk, and verification method.
- P0/P1 priorities improve trust, security, BI clarity, and daily action without UI or feature bloat.
- Changed code, if any, is covered by focused tests/gates/screenshots appropriate to the risk.
- Reports and evidence are saved in the repo, with paths named clearly.
- Remaining blockers are explicit, dated, and assigned to a next action.

### Non-Goals

- Do not rewrite the whole platform.
- Do not add new dashboards unless an existing route cannot support the workflow.
- Do not chase cosmetic polish before trust, correctness, and route consistency.
- Do not replace the existing stack or design language.
- Do not create speculative AI features before source data, permissions, and evidence contracts are reliable.
- Do not treat a passing redirect smoke test as authenticated UI quality proof.
